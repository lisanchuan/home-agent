## 一、整体架构

```
用户
   ↓
家庭管家 Agent（主Agent）
   ↓
工作Agent | 学习Agent | 财务Agent | 生活Agent | 记忆中心
```

### 核心设计理念

| 理念 | 说明 |
|------|------|
| 主Agent = 中心协调层 | 任务分解 + 派发 + 汇总 |
| SubAgent = 各领域专家 | 并行工作，独立执行 |
| 记忆中心 = 全局状态视图 | 所有状态可追踪、可控 |
| 三大工程铁律 | 收敛>发散、扁平>嵌套、状态中心化 |

### 各Agent职责

| Agent | 核心功能 |
|-------|---------|
| 工作Agent | 日历管理、邮件处理、任务管理、文档助手 |
| 学习Agent | 学习规划、知识问答、技能追踪、资源推荐 |
| 财务Agent | 收支记录、预算管理、账单分析、投资追踪 |
| 生活Agent | 日程协调、购物清单、餐饮规划、健康管理、智能家居 |
| 记忆中心 | 家庭知识库、成员档案、偏好设置、历史决策 |

## 二、架构七大问题

| 问题 | 严重度 | 核心风险 | 解决思路 |
|------|--------|---------|---------|
| 主Agent单点瓶颈 | ⭐⭐⭐⭐⭐ | 响应慢/挂则全瘫 | 简单任务直路由SubAgent |
| 知识库冷启动 | ⭐⭐⭐⭐ | 用户等不了直接放弃 | 预设模板+渐进式学习 |
| Agent能力边界模糊 | ⭐⭐⭐ | 抢活/推活/重复 | 场景-Agent映射表 |
| 进化机制难落地 | ⭐⭐⭐ | 用户懒得配合 | 被动收集+保守进化 |
| 隐私与安全 | ⭐⭐⭐⭐ | 财务/健康数据泄露 | 本地优先+操作分级 |
| 长期维护成本 | ⭐⭐ | MEMORY膨胀/知识过期 | 定期压缩+过期机制 |
| 跨领域边界划分 | ⭐⭐⭐ | 新场景无对应Agent | 动态Agent发现 |

## 三、Agent专家化与进化系统

### 核心问题：Agent为何不够专业

- 知识浅（通用模型不懂家庭场景）
- 不积累（每次从零开始）
- 不进化（犯过的错下次还犯）
- 缺反馈（不知道回答好不好）

### 专家化四步路径

1. 注入知识（RAG）
2. 专项训练（Prompt/RAG/Fine-tuning）
3. 实战打磨（反馈闭环）
4. 持续进化（L1-L4层次）

### 反馈三层机制

| 层次 | 方式 |
|------|------|
| 即时反馈层 | 用户👍👎评分 |
| 周期复盘层 | 日/周/月汇总分析 |
| 深度评审层 | 定期人工审核 |

### 进化四层次

| 层次 | 内容 | 触发 |
|------|------|------|
| L1 | 记忆进化 | 每次交互 |
| L2 | 知识进化 | 每周汇总 |
| L3 | Prompt进化 | 每月复盘 |
| L4 | 模型进化 | 积累足够数据后 |

### 三层记忆体系

| 系统 | 类型 | 说明 |
|------|------|------|
| System 1 | 短期 | 本次对话 |
| System 2 | 中期 | 当前任务上下文 |
| System 3 | 长期 | 家庭知识库RAG |

## 四、关键技术参考

### Multi-Agent动态派生架构（鹏宇AI大模型）

**核心运转逻辑：**
```
主Agent生成动态大纲 → delegate_task 工具派生 Subagent → Subagent并行工作
```

**强制压缩机制：** 子Agent回报时必须自我压缩，只带回提炼后的结论和事实

**三大工程铁律：**
1. 收敛 > 发散 — 不要让Agent无限发散
2. 扁平 > 嵌套 — 层级少一点，中心化控制
3. 状态中心化 — 所有状态必须可追踪、可控

### Agent vs Workflow

| | Agent | Workflow |
|--|-------|----------|
| 控制权 | Agent自主决定 | 人工预设流程 |
| 灵活性 | 高，可应对未知 | 低，适合固定流程 |
| 可靠性 | 不稳定 | 稳定可控 |

**行业趋势：Agentic Workflow** = Workflow可靠性 + Agent灵活性

## 五、做事规范（Superpower风格）

### 开发流程

```
1️⃣ brainstorming（理解需求）
       ↓
2️⃣ writing-plans（拆解任务）
       ↓
3️⃣ subagent-driven-development（执行）
       ↓
4️⃣ finishing-a-development-branch（收尾）
```

### 核心规则

| 阶段 | 规则 |
|------|------|
| brainstorming | 不理解清楚不动手；一次一问；用户批准设计后才进入下一步 |
| writing-plans | 任务粒度2-5分钟；每步有验证方法 |
| subagent-driven-development | 两阶段审查（设计合规 + 代码质量） |
| finishing-a-development-branch | 测试通过 + 文档更新 + 用户验收 |

### 核心原则

| 原则 | 含义 |
|------|------|
| YAGNI | You Aren't Gonna Need It — 不要做过度设计 |
| DRY | Don't Repeat Yourself — 不要重复 |
| TDD | Test-Driven Development — 先写测试再写代码 |
| HARD-GATE | 没有设计批准，绝对不开始写代码 |

### 触发条件

- ✅ 写代码/改功能 → 全流程
- ✅ 架构设计/方案讨论 → brainstorming + plans
- ✅ 修复Bug → debugging + plans
- ❌ 调研/探索 → 轻量级，直接做

## 八、关键结论

1. **网上没有成熟的家庭智能体系统** — 这是机会
2. **最佳路径**：OpenClaw底座 + 自研家庭Agent Skill模板 + 渐进式进化
3. **最难问题**：知识库冷启动 + 长期进化维护
4. **最终目标**：从"通用助手"变成"懂你家的专家"
