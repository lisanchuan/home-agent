# 知识冲突处理

> 当新信息和旧知识矛盾时的处理策略

---

## 一、冲突类型

### 1.1 事实冲突

两个事实不能同时为真

```
旧知识：用户住在北京
新知识：用户说"我现在在上海定居了"

判断：地点冲突，需要确认
```

### 1.2 偏好冲突

偏好随时间改变，并非真正的"冲突"

```
旧知识：用户喜欢川菜
新知识：用户说"最近胃不好，吃不了辣的了"

判断：这是偏好变化，不是冲突
处理：保留旧偏好，标注"已变更"，记录新偏好
```

### 1.3 习惯冲突

行为和预期不符

```
旧知识：用户每周五健身
新知识：用户连续3个周五都说要加班

判断：习惯可能被打破，需要观察
处理：标记"可能已变"，继续观察
```

### 1.4 关系冲突

涉及人际关系的变化

```
旧知识：用户说"我和女朋友准备结婚了"
新知识：用户说"最近单身了"

判断：关系状态冲突
处理：保留两者，标注时间范围
```

## 二、保守策略：观察3次原则

### 2.1 触发条件

当新知识与旧知识冲突时，进入"观察模式"

```
观察模式：
1. 不立即覆盖旧知识
2. 记录新知识的出现次数
3. 同时保留新旧知识
4. 3次出现后做出判断
```

### 2.2 决策树

```
           新信息出现
               │
               ▼
        ┌──────────────┐
        │ 与旧知识冲突？ │
        └──────────────┘
              │
         Yes  │  No
         ┌────┴────┐
         ▼         ▼
    进入观察    正常更新
         │
         ▼
    ┌────────────────┐
    │ 记录新知识      │
    │ 触发计数器+1    │
    └────────────────┘
         │
         ▼
    ┌──────────────┐
    │ 计数器 >= 3？ │
    └──────────────┘
         │
    Yes  │  No
    ┌────┴────┐
    ▼         ▼
 升级判断   继续观察
    │
    ▼
┌──────────────────┐
│ 3次中：           │
│ - >=2次支持新知识 → 新替旧│
│ - <=1次支持新知识 → 保留旧│
│ - 支持率相同 → 保留两者  │
└──────────────────┘
```

## 三、知识版本管理

### 3.1 版本记录

每个知识节点保留历史版本

```json
{
  "id": "pref_diet_spicy",
  "current": {
    "value": "不吃辣",
    "confidence": 0.95,
    "since": "2026-04-09",
    "source": "user_confirmed"
  },
  "history": [
    {
      "value": "喜欢吃辣",
      "from": "2026-01-01",
      "to": "2026-04-08",
      "reason": "用户反馈"
    }
  ]
}
```

### 3.2 回滚能力

- 任何时候可回滚到历史版本
- 回滚后保留回滚记录
- 用户可查看完整变更历史

## 四、冲突解决后的处理

### 4.1 新替旧

```
操作：
1. 将旧知识移入history
2. 新知识成为current
3. 记录变更时间和原因
4. 通知用户（可选）
```

### 4.2 保留旧知识

```
操作：
1. 标记旧知识为"被新知识覆盖"
2. 新知识正常写入
3. 如果未来新知识被否定，可回滚
```

### 4.3 保留两者

```
操作：
1. 旧知识添加时间范围限制
2. 新知识添加时间范围限制
3. 根据时间上下文选择使用哪个
```

```
示例：
旧知识："用户喜欢吃川菜"（2025年-2026年1月）
新知识："用户现在吃不了辣"（2026年2月起）

使用逻辑：
- 讨论2026年以前的餐厅推荐 → 用旧知识
- 讨论当前的饮食建议 → 用新知识
```

## 五、冲突检测的边界情况

### 5.1 近义词冲突

```
旧知识："用户住在朝阳区"
新知识："用户说他在北京"

判断：不冲突，北京包含朝阳区
处理：正常更新，更新地点信息
```

### 5.2 包含关系冲突

```
旧知识："用户每周运动3次"
新知识："用户每周运动5次"

判断：不完全冲突，都是运动
处理：更新频次，不触发冲突机制
```

### 5.3 时间范围冲突

```
旧知识："用户周五6点下班"
新知识："用户这周五加班到9点"

判断：下班时间是习惯，加班是例外
处理：保留习惯，标记"可能有例外"
```

## 六、冲突处理的用户体验

### 6.1 静默处理（无通知）

适用：低重要性、不影响日常的冲突

```
示例：用户改了健身时间
处理：静默更新，不打扰用户
```

### 6.2 事后通知

适用：中等重要性、有参考价值的变更

```
示例："注意到你最近加班比较晚，工作强度有变化吗？"
```

### 6.3 即时确认

适用：高重要性（健康、财务、关系）

```
示例："你之前说住在北京，现在说定居上海了，是搬家了吗？"
```
